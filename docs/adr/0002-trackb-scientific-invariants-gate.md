# 0002 - Track-B Scientific-Invariants Gate (vendored schema-spine)

## Status
Accepted (drafted 2026-06-25)

## Context
aop-mcp exposes a released read surface of `*.response` contracts under
`docs/contracts/schemas/read/`. One of them, `assess_aop_confidence.response`, is an
**AOP confidence assessment**: a deterministic heuristic over the AOP-Wiki RDF export
that produces a four-rung confidence call (`sparse_evidence` / `low` / `moderate` /
`high`) plus narrative `rationale` / `limitations` and OECD-dimension coverage. It is
interpretive *mechanistic context* â€” emphatically **not** a regulatory determination
and **not** an establishment of KER truth or causality about any chemical.

The NGRA platform's Track-B layer adds, per repo, a **scientific-invariants gate**: it
projects a released scientific object onto the canonical ToxMCP schema-spine and runs
the digest-pinned spine policy engine over it, BLOCKING if a future change ever lets a
scientific-overclaim regression reach a released object. This ADR records aop-mcp's
gate.

## Decision

### Released object + projection
The gate's corpus is the authentic, producer-emitted `assess_aop_confidence.response`
(regenerated by `scripts/build_spine_projection_goldens.py` running the real producer
over the `StubWikiAdapter`, `AOP:232`). It is projected â€” from DECLARED fields only â€”
onto two spine objects (`governance/project_to_spine.py`):

1. **`ClaimRecord`** at `claimClass = context_only`. `claimText` is assembled from the
   DECLARED narrative surfaces (`aop.title` / `aop.abstract` / `aop.evidence_summary`,
   `rationale[]`, `limitations[]`). `supportLevel` is mapped FAITHFULLY from the
   `overall_call` ladder, capped within the context-only band (never above `weak`):
   an AOP confidence call, however high, is interpretive context, not a decision-grade
   conclusion. The downstream-use posture authorizes only non-decision, mechanistic
   uses and explicitly prohibits causal / KER / adversity / risk / regulatory uses.
   `notARegulatoryConclusion = true`.
2. **`EvidenceAnchor`** at `evidenceClass = aop`, `role = context`,
   `supportDirection = context_only`, with `targetEntity` the AOP identity. This is the
   AOP confidence assessment AS A CONTEXT HANDOFF â€” never KER truth or primary causal
   evidence.

The claim is held at `context_only` and is NEVER inflated to `association` / causal /
risk / regulatory â€” postures the producer does not assert. This is the anti-overclaim
anchor: **an AOP confidence assessment is not a regulatory determination, and an AOP
context handoff is not KER or causal evidence.**

### Advertised codes â€” maximal-honest, each proven on a producer-contract-valid fault
Every advertised code BITES on a PRODUCER-CONTRACT-VALID source fault (one that passes
the strict source-contract guard, i.e. is something the real producer could emit), proven
through the REAL vendored bridge in
`tests/governance/test_scientific_invariants_adversarial.py`:

| Code | Driven by (DECLARED field) |
| --- | --- |
| `FREE_TEXT_OVERCLAIM` | safety overclaim in `rationale` / `limitations` / `aop.title` / `aop.abstract` |
| `ABSOLUTE_OR_REGULATORY_OVERCLAIM` | safety / regulatory-acceptance overclaim in the narrative |
| `CONTEXT_ONLY_OVERCLAIM` | causal / adverse / risk / safe assertion in the narrative |
| `AOP_CONTEXT_NOT_KER_EVIDENCE` | ker / causal language in `aop.title` flowing into the anchor's `targetEntity` |

A producer-emission-contract guard (`governance/source_contract.py`) runs at the TOP of
the gate, BEFORE any projection. It validates the raw object against the STRICT emission
contract (`governance/contracts/assess_aop_confidence.response.strict.schema.json`, an
`additionalProperties:false` tightening of the released response schema). Any object
carrying a schema-forbidden / undeclared field, or an out-of-enum `overall_call`, is a
`SOURCE_CONTRACT_VIOLATION` and is NEVER projected. This closes the
producer-emission-contract dead-arm class: a "fault" that could only fire a code by
carrying a field the producer cannot emit is caught here as a contract violation instead.

The strict contract was verified by running the REAL producer across two faithful paths
(a rich `StubWikiAdapter` emission and a fixture-fallback emission) so it is not authored
from a single sample. The two producer-OPTIONAL fields (`applicability_summary`,
`mechanism_role_summary`) are kept OPTIONAL â€” omitting one must not falsely reject a
faithful emission (the over-tighten trap).

### AI-provenance arm â€” HONEST-DROPPED (deterministic / side-channel)
aop-mcp's confidence assessment is DETERMINISTIC and zero-LLM: `overall_call ==
heuristic_overall_call`, provenance `phase1_oecd_alignment_normalization`, and NO
released `*.response` schema declares an AI / extraction / agent-provenance field. The
repo does carry an agent lane â€” `src/agent/workflows.py` + `docs/adr/agent-workflow-notes.md`
â€” but it is a **dry-run publish-plan orchestrator**: `WorkflowFactory.build_publish_workflow`
chains deterministic MediaWiki/OWL publish *planners* and an in-memory job submission;
its own ADR lists "integrate actual LangGraph" and any LLM step as *future work / open
questions*. No agent/LLM output enters a released object (`assess_aop_confidence.response`
or `export_draft_review_artifact.response` carry only deterministic, provenance-stamped
content). The AI-provenance arm is therefore a **side-channel**, and the gate projects no
`AssessmentRun` and advertises no spine AI code. If the agent lane ever begins feeding
LLM-derived content into a released object, the released schema must declare the
AI-provenance field and this gate's AI arm must be made live.

### Structural guardrails â€” ENFORCED but NOT advertised
The `context_only` claim ALSO satisfies, on every faithful emission, three structural
spine invariants the engine still runs over each projected object:
`ONTOLOGY_CONFIDENCE_CEILING_EXCEEDED` (support capped `<= weak`),
`CONTEXT_ONLY_BAD_DOWNSTREAM_USE` and `CONTEXT_ONLY_PROHIBITIONS_REQUIRED` (the
allowed/prohibited downstream-use sets are FIXED structural posture, not derived from any
declared source field). Because no producer-contract-valid SOURCE fault can drive them
through the faithful projection, advertising them would be an advertised-but-source-dead
arm; they are deliberately NOT advertised. They remain ENFORCED â€” a projection regression
that loosened the support cap or the downstream-use sets would still fire them and BLOCK.
The HIGH-CLAIM review codes dispatch only on a HIGH-STAKES `claimClass` the producer never
asserts and are likewise honest-dropped.

### Vendored engine + fail-closed bridge
The spine policy engine is vendored, digest-pinned, under `vendor/schema-spine/` at
`gitSha e0a6a0581efd8dfd5b10c2de14435d87769c5944` (59 files; `VENDORED_FROM.json` records
a sha256 per file). `scripts/vendor_verify.py` recomputes every digest and hard-fails on
tamper / untracked / missing files. The Python bridge (`governance/spine_bridge.py`) shells
to the vendored `run-policy.mjs` and is fail-closed by construction: missing Node /
non-zero exit / empty-or-unparseable stdout / timeout â†’ `ENGINE_UNAVAILABLE`; a tampered
vendored file â†’ `VENDOR_DIGEST_MISMATCH` (checked BEFORE the engine runs); a projected
object whose `schemaId` is not in the engine's recognized set â†’ `UNRECOGNIZED_SPINE_SCHEMA_ID`
(closing the engine's unknown-id `valid:true` no-op); an unmappable projection â†’
`PROJECTION_INCOMPLETE`. Every meta code BLOCKS; none is downgraded to a pass.

### CI
A new advisory `scientific-invariants` job in `.github/workflows/ci.yml` sets up Node,
runs `vendor:verify`, the governance test suite, then the gate. The existing `test` job
gains Node so `tests/governance/` runs there too. The gate is ADVISORY (no
required-status-check on the free-plan repo); promote it to a required check when the repo
gains branch protection / rulesets â€” the gate already exits non-zero on any blocking code.

## Consequences
- The gate is GREEN on the pristine corpus (the producer natively issues a non-overclaim,
  context-only conclusion); its job is to BLOCK a future regression.
- It is ADDITIVE: it touches no producer `src/` and no released contract, so the existing
  schema-validation / smoke / read-regression gates are unchanged. `aop-mcp` is a PUBLIC
  repo; this change does not alter repo visibility or settings.
- Promote-to-blocking requires only flipping the CI job to a required status check.
